# Plan: Further Performance Optimization for cross-platform-message-sync

## Current State Summary

The plugin has already undergone one major optimization round (Python → TypeScript for message sync core), achieving ~0.04ms avg latency and 2.6M msg/sec throughput. However, there are still significant efficiency opportunities in the remaining tools and infrastructure.

## Identified Optimization Opportunities

### Critical: 6 tools still spawn Python processes per call

**Impact: HIGH** — Each `spawn("python3", ...)` costs ~50-200ms in process creation overhead.

These tools still call `runPythonScript()` (python-runner.ts), spawning a new Python child process on every invocation:

| Tool | File | Python Script |
|------|------|---------------|
| `prompt_history_list` | `prompt-history-tool.ts` | `prompt_history.py` |
| `prompt_history_search` | `prompt-history-tool.ts` | `prompt_history.py` |
| `prompt_history_get` | `prompt-history-tool.ts` | `prompt_history.py` |
| `prompt_history_reuse` | `prompt-history-tool.ts` | `prompt_history.py` |
| `at_mention_status` | `at-mention-status-tool.ts` | `at_mention_router.py` |
| `code_copy_render` | `code-copy-tool.ts` | `code_render.py` |
| `quote_reply` | `quote-reply-tool.ts` | `message_quote.py` |
| `choice_select` | `choice-select-tool.ts` | `choice_render.py` |
| `choice_render` | `choice-select-tool.ts` | `choice_render.py` |

**Solution**: Rewrite each tool's core logic in pure TypeScript, same approach used for `OptimizedMessageSyncTool`. The Python scripts' logic is straightforward (JSON file reads, regex matching, string formatting) and trivially portable.

### Medium: python-runner-optimized.ts is unused dead code

`src/tools/python-runner-optimized.ts` implements a process pool but is never imported by any tool. All tools use `python-runner.ts` (simple spawn). This is dead code that should be removed.

### Medium: PerformanceMetrics uses O(n) slice instead of ring buffer

In `performance-monitor.ts:27-29`:
```ts
if (this.metrics.syncLatency.length > 100) {
  this.metrics.syncLatency = this.metrics.syncLatency.slice(-100);
}
```
Creates a new array every time the cap is hit. A fixed-size ring buffer would be O(1).

### Low: ConfigCache uses `watchFile` polling (5s interval)

In `optimized-message-sync.ts:68-71`, `watchFile` with 5000ms interval polls the filesystem. `fs.watch` (inotify-based) is more efficient for file change detection. However, this is low priority since the cache is only used for config files accessed infrequently.

### Low: LRU Cache in optimized-message-sync.ts is unused

The `LRUCache` class and `ConfigCache` class are defined in `optimized-message-sync.ts` but `ConfigCache` is never instantiated — the tool doesn't actually read config files during sync. The `LRUCache` and `ConfigCache` classes are dead code.

### Low: Duplicate Python implementations (TS and Python side-by-side)

The Python scripts in `tools/` duplicate logic already in TypeScript:
- `message_unify.py` ≈ `optimized-message-sync.ts` (parser/formatter logic)
- `code_render.py` — extract/render code blocks (also in `optimized-message-sync.ts:extractCodeBlocks`)

After rewriting tools to TypeScript, the Python scripts can be kept only for backwards compatibility / standalone CLI use.

### Low: Hooks do trivial logging with no real value

`src/hooks/index.ts` registers two hooks that only log tool call/completion. This adds overhead to every tool call with minimal value. Consider making hooks conditional on a debug flag.

## Plan Tasks

### Task 1: Rewrite prompt-history tools in TypeScript
Port `prompt_history.py` logic to TypeScript:
- Create `src/tools/prompt-history-native.ts` with native TS implementation
- JSON file read/write using `fs/promises` (async, non-blocking)
- Search with simple keyword scoring (same as Python version)
- Update `PromptHistoryListTool`, `PromptHistorySearchTool`, `PromptHistoryGetTool`, `PromptHistoryReuseTool` to use native implementation
- Delete `python-runner.ts` import from prompt-history-tool.ts

### Task 2: Rewrite code-copy-render tool in TypeScript
Port `code_render.py` logic to TypeScript:
- Code block extraction (regex-based, same pattern already exists in `optimized-message-sync.ts:extractCodeBlocks`)
- Platform-specific renderers (string templates)
- Update `CodeCopyRenderTool` to use native implementation

### Task 3: Rewrite choice-select/render tools in TypeScript
Port `choice_render.py` logic to TypeScript:
- Option extraction (regex patterns)
- Multi-select parsing (range expansion, Chinese ordinals, split patterns)
- Platform-specific render hints
- Update `ChoiceSelectTool` and `ChoiceRenderTool` to use native implementation

### Task 4: Rewrite quote-reply tool in TypeScript
Port `message_quote.py` logic to TypeScript:
- Quote parsing (regex patterns for standard, simple, reply-at, CQ formats)
- Quote header generation
- TopicTracker class (JSON file persistence)
- Update `QuoteReplyTool` to use native implementation

### Task 5: Rewrite at-mention-status tool in TypeScript
Port `at_mention_router.py` logic to TypeScript:
- MentionRouter class (alias index, rate limiting, trigger logic)
- Config loading from JSON
- Update `AtMentionStatusTool` to use native implementation

### Task 6: Clean up dead code
- Remove `src/tools/python-runner.ts` (no longer needed after all tools are native)
- Remove `src/tools/python-runner-optimized.ts` (was never used)
- Remove unused `LRUCache`, `ConfigCache` classes from `optimized-message-sync.ts`
- Remove `message-sync-tool.ts` (old Python-based sync tool, replaced by optimized version)

### Task 7: Optimize PerformanceMetrics ring buffer
- Replace array + slice with a fixed-size circular buffer for O(1) recording

### Task 8: Update tests
- Update existing tests to test native implementations instead of mocking `runPythonScript`
- Add unit tests for new native implementations
- Verify all tests pass

### Task 9: Run verification
- `npm run typecheck`
- `npm run lint`
- `npm run test`
