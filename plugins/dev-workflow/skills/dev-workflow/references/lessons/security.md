## grep 扫描硬编码密钥模式
- **来源**：freeapi 重构 (2026-05-06)
- **症状**：需要确认重构后代码中没有硬编码的 API Key
- **根因**：Python 项目中 API Key 管理容易混乱
- **方案**：使用 grep -rn "API_KEY\s*=" + grep -rn "SECRET\s*=" + grep -rn "PASSWORD\s*=" 扫描，配合 grep -v "os.getenv" 排除合法读取
- **关键要点**：安全扫描三件套：API_KEY、SECRET、PASSWORD grep。重构后必须跑一遍

## 静默吞异常隐藏安全问题
- **来源**：freeapi 重构 (2026-05-06)
- **症状**：except: pass 不仅影响调试，还可能隐藏安全事件（认证失败被吞、权限检查被跳过）
- **根因**：过度防御性编程
- **方案**：安全审计中扫描 `except.*:.*pass` 模式，每个实例必须说明为什么安全
- **关键要点**：安全审计 = 查密钥 + 查异常吞没 + 查输入验证

## 环境变量命名一致性
- **来源**：freeapi 重构 (2026-05-06)
- **症状**：同一个 Provider 的 API Key 在不同文件中用不同名字读取（如 CODINGPLAN_API_KEY vs CODINGPLAN_GLM_API_KEY）
- **根因**：缺少统一的配置文件，各模块独立命名
- **方案**：集中配置文件定义所有环境变量名，统一前缀（如 PROVIDER_NAME_API_KEY）
- **关键要点**：环境变量命名 = PROVIDER_ + 资源类型。集中定义，散落读取是安全隐患

## 死代码可能隐藏安全漏洞
- **来源**：freeapi 重构 (2026-05-06)
- **症状**：middleware.py 中存在完整的 CORS 中间件实现但从未被使用（死代码），如果被误激活可能暴露不安全的 CORS 配置
- **根因**：重构过程中遗留的未使用代码
- **方案**：安全审计时扫描未使用的类和函数，删除死代码。用 Python 的 coverage.py 识别未覆盖代码
- **关键要点**：死代码 = 潜在攻击面。最小化原则，不需要的代码删掉
